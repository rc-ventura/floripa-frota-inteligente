## Tech Leader Verdict

### Architecture Assessment
The 12-table schema is well-built and materializes `data-model.md` faithfully. Naming convention centralized (`_NAMING` on `Base.metadata`, models.py:16), single URL resolution point (`db/config.py` — no layer invents its own connection, honoring constitution VI + research R1), FK PRAGMA wired per-connection for SQLite (config.py:27), staging deliberately loose (TEXT-nullable, no FK/CHECK), and the alerta partial unique index is genuinely elegant (research R6). SOLID/robustness at schema level is sound: LIMIAR_CONFIG is the single source of thresholds seeded from one JSON (no business constants in code — constitution V), `fonte_origem` on every consolidated table incl. veiculo (R8), idempotency proven for migration + seed.

**The one real architectural issue is a self-inconsistency in how "upsert key" handles NULL — verified empirically, not just by reading the QA report.**

Reproduced the exact three keys in raw SQLite:
- `abastecimento(placa,data,km_hodometro)` — two rows same (placa,data) with NULL km → **both inserted** (constraint silent). Same rows with a non-null km → 2nd correctly rejected.
- `multa(placa,data,valor,condutor_pseudo)` — two rows with NULL condutor_pseudo → **both inserted**. With a non-null condutor → 2nd correctly rejected.
- `manutencao(placa,data,tipo)` — **rejected correctly.** Its three key columns are all NOT NULL (models.py:80-82), so manutencao does NOT have this bug — it's only a test-coverage gap. (QA lumped it into "the three untested keys"; the correctness defect is abastecimento + multa only.)

The NULL-slip is real and confirmed — standard SQL `NULL≠NULL` in UNIQUE, true in both SQLite and Postgres by default. `alerta` sidesteps it with `coalesce(limiar_id,-1)` (models.py:144) — the team clearly knows the pattern — but did not apply it to abastecimento/multa.

**Independent judgment on whether it contradicts the spec (differs from QA's flat framing):**
- **abastecimento:** research R7 (research.md:138) explicitly says *"NULLs de km não conflitam — dedup fina fica no pipeline."* The NULL-slip here is a conscious, documented decision, not an accident, and may even be correct: two same-day fill-ups with unknown km could be two real events; forcing a coalesce collision would wrongly reject a legit 2nd fill-up. The gap is that the contract (`esquema_tabelas.md:29`) never restates this caveat — its "Limitação aceita da PoC" clause only mentions multas.
- **multa:** the sharper problem. The contract makes an unqualified promise — *"duas multas legítimas idênticas no mesmo dia colidem na chave — a 2ª vai para log_qualidade com motivo duplicado"* (esquema_tabelas.md:29) — that the schema does not deliver when condutor_pseudo is NULL. Research R7 doesn't flag it. This is a genuine contract-vs-implementation lie, and it feeds the idempotency constitution principle + the demo-critical custos panel (double-counted fines).

Not a code deviation — the code implements data-model.md exactly (data-model.md:44,71 literally specify nullable-column UNIQUE keys). A spec/contract-level design flaw the code faithfully inherited. Severity: MEDIUM at this boundary (spec 002 is schema-only → latent, not triggered), but it becomes active the moment spec 003's Load step trusts these keys, and spec 003 is next.

### ADR Compliance
- ✅ **ADR-001 (placa canônica dual):** Compliant. Regex `^[A-Z]{3}[0-9][A-Z0-9][0-9]{2}$` (models.py:13) is character-for-character equivalent to ADR-001's `^[A-Z]{3}\d[A-Z\d]\d{2}$`. Applied via `@validates` + CHECK length=7 backstop; tested for both formats and 4 malformed rejections (test_db.py:94).
- ✅ **ADR-002 (persistir km_hodometro em abastecimento):** Compliant. Column present, Integer nullable, migration line 108, existence-tested. *Tension:* ADR-002 mandates the nullability ("registro sem km continua válido") that directly creates the NULL-slip in the upsert key. ADR-002 and R7 half-acknowledge it but never reconciled it against the contract's promise — root of the issue.
- ✅ **ADR-003 (categoria preventiva/corretiva + calibração):** Compliant. `categoria` NOT NULL with CHECK; troca_oleo/revisao_geral vocab matches.
- 📝 **New ADR needed:** *"Tratamento de NULL em chaves de unicidade/upsert das consolidadas."* Right now the schema has three NULL-in-key situations resolved three inconsistent ways — alerta (coalesce sentinel), abastecimento (NULL-slip, pipeline-side per R7), multa (NULL-slip, undocumented). A future contributor cannot tell which are deliberate. Should (a) codify the coalesce-to-sentinel pattern as project convention, (b) record the per-table decision and why they differ, (c) bind the rule "any nullable column in an upsert key must either coalesce-to-sentinel OR have its NULL-dedup responsibility explicitly assigned to the pipeline in the contract." Will recur in specs 003/004.

### Learning Lessons Applied
- ⚠️ **"Proteja a âncora do ruído: teste o invariante negativo"** — Partially applied; this issue is a textbook recurrence of its second lesson. Positive/constructed invariants are tested richly — including the alerta NULL/coalesce collision (test_db.py:177-183), proving the author knew NULL-collision matters. But the negative invariant "two duplicate abastecimentos/multas with a NULL key must not both insert" has zero test — nobody constructed it because it's an absence. The lesson's meta-rule — "the protected set is derived from consumer semantics, not a list of special cases" — is the same slip: alerta derived its key from the motor's "1 active alert" semantics; abastecimento/multa special-cased NULL ad hoc.

### Cross-Report Synthesis
QA and Security independently converge (both PASS, both corroborated by Devin): branch is fundamentally sound. LGPD posture clean and enforced by an introspection test, no secrets, no injection, deps current. The single cross-cutting theme worth escalating is the NULL-in-upsert-key gap, which touches all three lenses at once: QA (correctness + missing negative tests), Security (only tangentially), Architecture (contract lies to the next consumer + idempotency principle at risk). Isolated in blast radius (only NULL-key rows) but systemic in origin (no project convention for NULL-in-key) — hence the ADR. Security's items (redact DATABASE_URL print, `text()`-wrap PRAGMA, cnh-in-registro_bruto) are all correctly non-blocking and mostly belong to spec 003/007.
